# AI 协同开发日志

## 1. 工具链

- AI 工具：Codex / ChatGPT
- 前端：Vite、React、TypeScript、Tailwind CSS、Recharts、Phosphor Icons
- 后端：FastAPI、SQLite、Pydantic、httpx
- 测试：pytest
- 版本管理：Git 本地提交，稳定后推送 GitHub

## 2. 需求拆解记录

### 原始任务

选拔指南要求“口袋记账 AI 版”至少包含基础记账、流水统计，并实现一句话记账或预算毒舌看板等进阶能力。

### AI 协作判断

我让 AI 先读取选拔指南，再根据我的技术基础和作品观感要求拆解方案。第一轮 AI 建议使用 Streamlit，但我认为 Streamlit 画面容易显得低级，不适合作为选拔作品。随后我们调整为前后端分离方案：

- 前端使用 Vite + React + TypeScript + Tailwind
- 后端使用 FastAPI + SQLite
- AI 只做解析和建议，不直接改数据库
- 金额以整数分存储，回应指南中的浮点数精度踩坑点

采纳原因：这个方案既能体现 Vibe Coding 能力，又能保证代码结构可解释。

```text
请根据选拔指南、我的技术基础和答辩需要，判断口袋记账 AI 版应该使用什么技术栈。
```

### 结果

AI 给出了“Vite + React + TypeScript + FastAPI + SQLite”的前后端分离方案，并把原生 App、多用户、PWA、银行同步、OCR 发票等功能排除到第一版之外。

## 4. 核心 Prompt 2：一句话记账结构化解析

```text
请把中文自然语言记账句子解析成固定 JSON。
字段必须包括：
amount_cents, type, category, account, occurred_at, note, confidence, missing_fields。
如果字段缺失，不要编造，写入 missing_fields。
AI 不能直接入账，只返回待确认数据。
```

### 结果

后端实现了 OpenAI 兼容接口调用，同时保留本地规则解析作为兜底。即使 API Key 缺失或模型调用失败，常见句子仍可解析，演示流程不会中断。

## 5. Debug 经历

### 严重 Bug：货币浮点精度风险

问题：如果前端或后端直接用浮点数保存金额，`0.1 + 0.2` 会得到 `0.30000000000000004`，预算统计可能出现误差。

排查方式：

1. 将金额处理逻辑单独抽成 `parse_yuan_to_cents`。
2. 用 pytest 写行为测试，验证 `0.1` 和 `0.2` 分别转为 10 分和 20 分，相加得到 30 分。
3. 检查数据库 schema，确认账单金额字段为 `amount_cents INTEGER`。

修复：

- 后端使用 `Decimal` 解析金额字符串。
- 数据库只保存整数分。
- 前端也通过字符串解析金额，提交给后端前转成整数分。

回归测试：

- `0.1`
- `0.2`
- `19.9`
- `50元`
- `今天中午和室友吃疯狂星期四花了 50 块，微信付的`

## 6. 采纳与拒绝记录

- 采纳：前后端分离，因为更像完整工程，也更能体现 Vibe Coding。
- 采纳：SQLite，因为单用户演示足够稳定，答辩解释成本低。
- 采纳：预算毒舌看板，因为它同时展示图表能力和 AI 反馈能力。
- 拒绝：Streamlit，因为作品观感容易像数据小工具。
- 拒绝：原生 App，因为 20 天内会显著增加解释和调试成本。
- 拒绝：多用户登录，因为第一版重点不是权限系统。

## 7. 第二轮协作：页面和功能增强

### 用户反馈

第一版功能能跑通，但前端仍然太朴素，缺少设计感；流水、图表、设置和 AI 解析细节也不够明确。

### Skill 选择

- 使用 `design-taste-frontend`：作为反低质前端检查表，约束不要出现泛 AI 紫色、默认蓝白后台、三等分卡片堆砌、无文字解释的图表。
- 使用 `impeccable`：作为产品 UI polish 依据，重点检查信息层级、设置面板、表单状态、响应式和可解释性。
- 未使用游戏、PPT、生成艺术类 skill：这些和本项目的任务型产品界面不匹配。

### 第二轮 Prompt

```text
现在的前端页面还是太朴素，也没有设计感。请调用本机可用的前端和设计 skill 优化页面。
功能上要实现：
1. 流水列表按照日期分割。
2. 做出自定义标签功能。
3. 图表里要有文字信息，清晰呈现。
4. 设置部分要显式做出来，放 API 配置，按 OpenAI 兼容接口字段设计。
5. 把这些要求写进 AI 协作日志，并报告 AI 快记、本地兜底和置信度实现细节。
```

### 实现结果

- 前端增加顶部状态带：净现金流、预算线、AI 接口状态。
- 消费趋势图增加文字 insight：最高分类、主要账户、日均支出、预算使用率。
- 饼图增加图例文字：分类名、占比和金额直接可见。
- 流水列表改为日期分组：每个日期组显示当天笔数、支出、收入和净额。
- 交易表单增加自定义标签：标签可添加、去重、删除，并随交易一起保存。
- 设置面板显式展示：Base URL、Model、API Key 输入位、`.env` 预览和后端非敏感配置状态。
- 后端新增 `/api/settings/public`：只返回 Base URL、模型名、是否配置 Key、数据库文件名，不泄露真实密钥。

### AI 快记细节

当前 AI 快记有两条路径：

1. 有 API Key 且模型配置有效时，后端调用 OpenAI 兼容 Chat Completions 接口，要求模型返回固定 JSON。
2. 无 API Key 或模型调用失败时，后端进入本地兜底。

本地兜底是确定性规则，不是黑盒模型：

- 金额：用正则识别句子里的数字金额，再用 `Decimal` 转成整数分。
- 分类：用 `CATEGORY_KEYWORDS` 关键词字典匹配，例如“疯狂星期四、咖啡、外卖”归入餐饮。
- 账户：用 `ACCOUNT_KEYWORDS` 匹配“微信、支付宝、银行卡、现金”。
- 类型：收入类关键词会标为 `income`，否则默认 `expense`。
- 备注：从原句去掉金额、时间词、账户词和常见动词，留下短备注。
- 置信度：字段完整为 `0.86`，缺金额、分类或账户时为 `0.52`，并写入 `missing_fields`。

设计原因：选拔项目更需要可解释、可维护和可演示，而不是把所有能力藏进模型调用里。

- 不做长页面下滑合集。
- 设置必须作为独立入口。
- 前端需要大改配色和布局。
- 如果需要更换技术栈，需要先说明并共同决策。

### Skill 和技术决策

- 使用 `impeccable`：检查任务型产品界面是否有清晰信息架构、稳定组件和可信工作台气质。
- 使用 `design-taste-frontend`：作为反低质前端检查表，重点排除模板后台、廉价渐变和信息堆叠。

### Debug 案例：月初首页统计为空

问题：2026-07-01 截图时，流水页有数据，但首页统计曾出现空态，影响演示第一印象。

排查：

1. 直接请求 `/api/transactions?month=2026-07` 和 `/api/stats/monthly?month=2026-07`。
2. 检查 SQLite 种子逻辑，发现旧演示数据大多落在 6 月；如果数据库已有任意交易，种子逻辑不会补当前月数据。
3. 修复为只检查当前月份是否已有交易；同时限制 `days_back` 不超过当前月已经过去的天数，避免 7 月 1 日减一天落到 6 月。

修复结果：

- 7 月统计接口能返回收入、支出、预算和分类明细。
- 页面首屏在当前月份有可演示数据。
- 这个案例可用于答辩说明：不仅做页面，还处理了演示数据、时间边界和可验证接口。

## 9. 第四轮协作：大理石质感与动效修正

### 用户反馈

第三轮虽然完成了页面切换，但截图中的总览主卡仍然像一整块黑红色大面板，不够高级；主卡上方还缺少一块控制信息；整体配色过于单一。用户希望改成更有质感的方向，例如大理石材质，并加入一些动画。

### 技术和设计决策

- 不更换前端技术栈：当前 React + CSS 足够完成材质、页面切换和轻量动效。
- 不新增动画库：动效只服务状态反馈，用 CSS 即可解释和维护。
- 视觉从“黑白高对比 + 朱红”改为“浅色大理石 + 烟灰玻璃 + 多角色矿物色”。
- 色彩不再单一：深海绿用于主操作和现金流，香槟金用于辅助状态，梅紫用于分类提示，低饱和红只用于错误或超支。

### 第四轮 Prompt

```text
太丑了，我图里的那部分上面还缺了一块控件，不要这样的配色。
不要做单一的配色，做得高级一点，大理石质感那种。
想一下哪里可以做点动画。
```

### 实现结果

- 总览主卡补充顶部控制条：预算使用率、流水笔数、接口状态。
- 主卡从暗色大块改为浅色大理石板，使用多层线性纹理模拟石材脉络。
- 左侧导航从黑色应用壳改成烟灰玻璃面板，降低压迫感。
- 全局背景加入缓慢漂移的大理石纹理，不使用额外图片资源，部署更简单。
- 图表色盘改为深海绿、香槟金、梅紫、矿物灰等多角色色。
- 动效包括页面模糊淡入、卡片轻浮起、按钮 hover 位移、预算环渐进和背景纹理慢速漂移。

### 答辩可解释点

这些动画都在 CSS 中实现，没有引入 GSAP、Motion 等额外依赖。理由是本项目重点是记账产品的可解释性和维护性，轻量 CSS 动效足够展示产品感，同时不会增加初学者答辩成本。

## 10. 第五轮协作：Agnes API 接通与相对日期修正

### 用户反馈

用户在网页端填写 API Key 后，AI 快记仍显示本地规则兜底，需要确认到底是接口没接上、配置文件格式不对，还是后端没有重启读取新环境变量。

### 排查过程

1. 检查 `/api/settings/public`，确认前端设置页只展示非敏感状态，真实后端配置来自 `backend/.env`。
2. 发现后端仍读到默认模型名 `your-model-name`，因此 `parse_with_model` 会主动走 `local_rule`。
3. 将 OpenAI 兼容配置改为 Agnes 的 `/v1` Base URL 和模型名后重启后端。
4. 再次请求 `POST /api/ai/parse-transaction`，返回 `source: "model"`，证明已经调用真实模型。

### Debug 案例：模型把“今天”解析成旧日期

问题：Agnes 接通后能返回结构化 JSON，但把“今天中午”解析成历史日期，影响记账准确性。

修复：

- Prompt 中加入当前本地时间，要求模型按当前时间换算今天、昨天、前天。
- 后端新增 `detect_occurred_at` 和 `has_relative_date`。
- 如果用户原句包含今天、昨天、前天，最终 `occurred_at` 采用后端本地确定性日期，而不是完全相信模型返回。
- 本地兜底解析也复用同一套相对日期规则。

### 当前结论

- Agnes 已通过 OpenAI 兼容接口接入。
- AI 快记的正常来源应显示为 `model`。
- 如果 Key、Base URL、模型名缺失或 API 报错，系统会返回 `local_rule` 或 `error_fallback`，不会阻塞手动记账。
- 当时的设置页不能真正保存后端密钥，真实密钥必须写在 `backend/.env` 并重启后端；这个限制在第六轮已改为后端真实保存。
- 本机 8000 端口存在异常残留监听，前端需要通过 `frontend/.env.local` 指向 `http://127.0.0.1:8001`，否则会打到未配置 Key 的旧后端。
- Agnes 偶发响应超过 15 秒，因此后端新增 `AI_REQUEST_TIMEOUT_SECONDS`，本机默认调到 45 秒；即便超时，系统也会进入 `error_fallback` 而不是让记账流程中断。

## 11. 第六轮协作：真实设置页与备用模型

### 用户反馈

用户认为备用 API Key 和备用模型是合理需求：如果主模型调用错误，应尝试备用模型。同时，设置页不应该只是浏览器本地演示，而应该真正影响后端运行配置。

### 技术决策

- 接受备用模型需求：这能提升演示稳定性，也能体现产品级容错意识。
- 不做云端密钥系统：第一版是本地单用户项目，没有登录和多用户隔离。
- `.env` 保留为默认值和部署兜底。
- 网页设置页保存后写入 SQLite 的 `app_settings` 表，后端下一次 AI 调用即时生效。
- API Key 不在公开设置接口中回显，只返回是否已配置。

### 实现结果

- 新增 `PUT /api/settings/ai`，保存主接口、备用接口、Key 和超时时间。
- `GET /api/settings/public` 增加主备接口状态、备用是否启用和超时时间。
- AI 解析和月度建议都改为主备模型顺序调用：主模型失败后尝试备用模型，主备都失败才进入 `error_fallback`。
- `ParseResult` 和 `AdviceResponse` 增加 `provider` 字段，能区分 `primary`、`backup`、`local` 和 `fallback`。
- 设置页从 localStorage 演示改成真实后端保存，并显示主 Key、备用 Key、SQLite 文件和备用启用状态。

### 测试与验证

- 新增测试：保存设置后不回显真实 Key。
- 新增测试：主模型超时后会调用备用模型，并返回 `provider: "backup"`。
- 后端测试 `10 passed`。
- 前端 `npm run build` 通过。

### Debug 案例：备用模型名错误

问题：设置页显示备用 Key 已配置，但临时关闭主接口后仍进入 `error_fallback`。

排查：

1. 直连硅基流动备用接口，发现 Key 可用，但模型名 `deepseekv4-flash` 返回 “Model does not exist”。
2. 调用 `/models` 列出当前 Key 可见模型，找到正确模型 ID：`deepseek-ai/DeepSeek-V4-Flash`。
3. 通过设置接口修正备用模型名后再次测试。

修复结果：

- 主接口临时指向不可连接地址时，AI 快记返回 `source: "model"`、`provider: "backup"`。
- 备用模型解析“今天晚上买书花了36块支付宝”，返回金额 3600、分类“学习”、账户“支付宝”。
- 发现备用模型可能返回 `books`、`alipay` 等英文/别名字段，因此后端新增分类和账户归一化：`books/书籍 -> 学习`，`alipay -> 支付宝`，`微信支付 -> 微信`。

## 12. 第七轮协作：阶段评估与主备模型可验证化

### 自主评估

当前项目已经具备完整记账闭环：快记、确认入账、流水、预算、图表、设置、AI 兜底和主备模型。继续加入语音输入、App、OCR 等大功能会提高复杂度，但不一定提高选拔评审对“代码可负责、AI 协作可解释”的信任。

因此本轮优先级不是继续堆功能，而是增强可验证性：让评审能在设置页直接测试主接口和备用接口是否真的可用。

### 本轮计划

- 后端新增主备模型连接测试接口。
- 设置页新增“测试主备接口”按钮和结果卡片。
- 测试结果展示 provider、模型名、通断、耗时和简短错误，不回显 API Key。
- 用行为测试保证接口不泄露密钥。
- 用前端构建和浏览器截图检查桌面/移动端可用性。

### 验收标准

- `POST /api/settings/ai/test` 能返回主/备接口测试结果。
- 测试接口不返回真实 Key。
- 设置页能展示“主模型/备用模型”的连接状态和耗时。
- 后端测试通过。
- 前端构建通过。
- 桌面和手机宽度无明显布局崩坏。

### 实现结果

- 新增 `AiProviderTestRequest` 和 `AiProviderTestResult`。
- 新增 `POST /api/settings/ai/test`。
- 设置页新增“测试主备接口”按钮。
- 设置页新增主/备模型测试结果卡片。
- 新增测试：测试接口返回主/备状态但不泄露 Key。
- 后端测试从 10 条增加到 11 条。

## 13. 第八轮协作：交付整理与部署边界

### 自主评估

项目已经具备核心产品闭环，下一步不应继续扩展语音输入、App、OCR 或银行账单导入。那些功能会显著增加解释和维护成本，但对本次选拔的核心评分点提升有限。

本轮更重要的是交付可信度：评审能否从 README 快速理解项目，能否本地跑起来，能否部署，能否看懂 AI 协作过程，能否确认仓库没有密钥和本机噪声。

### 本轮计划

- 把 README 改为评审入口。
- 补充 Vercel + Render 的部署路径。
- 增加演示脚本和发布检查表。
- 修复前后端分离部署的 CORS 边界。
- 准备 GitHub 发布，但不在未登录 `gh`、未配置 remote 时强行推送。
- 生成桌面截图作为 README 视觉证据。

### 关键实现

- 新增 `CORS_ALLOWED_ORIGINS`，后端从环境变量读取允许的前端域名。
- 新增 `render.yaml`，给出 Render 后端部署参考。
- 新增 `frontend/vercel.json`，处理 Vercel 静态站点刷新回退。
- 新增 `docs/DEMO_SCRIPT.md`，用于 3 分钟演示。
- 新增 `docs/RELEASE_CHECKLIST.md`，用于提交和部署前验收。
- `.gitignore` 增加本机设计工具目录和误生成的后端 `package-lock.json`。

### 决策理由

- 接受 CORS 环境变量：线上前端域名部署后会变化，不能继续写死本地地址。
- 不把本机 `.impeccable/` 目录放进仓库：它是本地设计工具运行配置，不是项目交付内容。
- 不把后端 `package-lock.json` 放进仓库：后端是 Python 项目，这个空 lockfile 是误生成噪声。
- 不在本轮强行推送 GitHub：本机 `gh` 未登录，且当前没有 remote。先把项目整理到可发布状态，再由用户完成登录或创建仓库后推送。

### 验收标准

- 后端测试通过。
- 前端构建通过。
- README 中截图文件真实存在。
- 仓库不包含真实 `.env`、数据库、`node_modules`、构建产物。
- 部署文档包含前端域名、后端域名和 CORS 配置关系。

## 14. 第九轮协作：AI 财务点评结构化

### 用户反馈

用户指出当前 AI 总结可能只有一句话，担心本地兜底只是内置几条语录后按权重挑选，并提出希望财务点评也明确走 AI：内置角色设定 Prompt，输出详细分析和一句话分析，两个前端板块共用同一套 AI API。

### 实际旧实现

旧版 `GET /api/ai/monthly-advice` 的模型 Prompt 只要求“输出一句 40 字以内中文建议”。本地兜底不是权重抽语录，而是按预算使用率做确定性 if 判断：未设置预算、已超支、接近上限、预算健康，各返回一句固定建议。

### 本轮修改

- 后端 `AdviceResponse` 增加 `headline`、`detail`、`action_items`。
- 模型 Prompt 增加角色设定：个人财务分析师、面向学生和年轻用户、基于统计、不编造数字、支持直接/温和两种语气。
- 模型输出改为固定 JSON：一句话结论、详细分析、行动建议。
- 本地兜底升级为结构化规则分析，基于预算使用率、最高分类、主要账户、日均支出和剩余额度生成同样字段。
- 前端总览页显示一句话结论和详细分析，预算页显示详细分析和行动建议。
- 前端把基础数据加载和 AI 建议加载拆开：流水、统计、预算先显示，模型慢或超时时只影响建议板块。
- `advice` 字段继续保留为一句话摘要，避免旧调用方式突然失效。

### 验收标准

- 无模型配置时，接口也返回 `headline/detail/action_items`。
- 模型返回 JSON 时，前端能使用结构化字段。
- 主备 Provider 机制不改变。
- 模型响应慢时，页面主体数据不被阻塞。
- 后端测试和前端构建通过。
