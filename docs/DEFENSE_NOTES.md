# 答辩备忘

## 项目一句话

口袋记账 AI 版把中文收支描述解析成可确认的结构化账单，用 SQLite 保存真实流水，再由同一套 OpenAI 兼容 Provider 分析本月预算和消费结构；模型失败时仍保留本地规则与手动入账。

## 为什么前后端分离

React 负责页面、表单和状态反馈；FastAPI 负责数据校验、AI 调用、统计和数据库。API Key 只进入后端，前端看不到真实密钥。相比把所有代码放在一个脚本里，这种结构更容易说明每层责任。

## 为什么使用 SQLite

SQLite 是嵌入式关系数据库，数据保存在一个 `.db` 文件中，不需要额外启动数据库服务。单用户演示不需要 MySQL 或云数据库的运维成本，但仍然具备真实持久化、查询和约束。

本地数据库可长期保留。Render 免费实例没有持久磁盘时，重新部署或实例重建可能重置文件，因此线上定位为演示环境。

## 为什么金额保存整数分

浮点数不能可靠表示所有十进制小数。系统把 50 元保存为 5000 分，把 19.9 元保存为 1990 分：

```text
输入字符串 → Decimal/字符串拆分 → 四舍五入到整数分 → SQLite INTEGER
```

统计时只对整数求和，展示时再除以 100，因此不会出现 `0.1 + 0.2 = 0.30000000000000004`。

创建和更新交易只接受 `1–9,999,999,999` 分，也就是 `0.01–99,999,999.99` 元。AI 未识别金额时，解析草稿可以暂时使用 0 并标记 `missing_fields`，但交易接口和 SQLite 触发器都不允许把 0 元写成正式流水。

## 交易时间为什么统一为北京时间

产品面向中国用户，Render 服务器本身可能运行在 UTC。如果直接使用无时区 `datetime.now()`，北京时间凌晨的“今天”可能仍被服务器算成前一天。

后端把业务时区固定为 `Asia/Shanghai`，所有进入交易接口的时间都会归一为带 `+08:00` 的 ISO 8601。无时区历史值按北京时间解释，UTC 或其他偏移会转换到北京时间。月度筛选、周一到周日统计和日期分组都从业务时区重新取日期，不截取服务器 UTC 字符串。

前端 `datetime-local` 不携带时区，因此使用专门的双向转换函数：API 的 `+08:00` 转成可编辑的 `YYYY-MM-DDTHH:mm`，提交时再明确补回 `+08:00`。没有用 `toISOString()` 保存表单时间，因为它会转换成 UTC，容易让日期少一天。

## AI 快记完整流程

1. 前端发送原始中文到 `POST /api/ai/parse-transaction`。
2. 后端先调用主 OpenAI 兼容接口，当前线上为 Groq `qwen/qwen3.6-27b`。
3. 完整配置备用 Provider 时，主模型超时、网络失败、JSON 错误或字段校验失败后继续尝试备用模型。
4. 没有备用配置或所有 Provider 都失败后，进入确定性本地规则。
5. 模型字段归一化到系统允许的分类和账户。
6. Pydantic 校验金额、类型、日期、长度和标签。
7. 前端显示真实来源和缺失字段；置信度保留在接口数据中，不用一个看似精确的百分比替代用户核对。
8. 用户确认后才调用交易写入接口。

AI 不直接入账，因为模型可能误解金额、日期或账户。用户确认是数据正确性的最后边界。

解析成功时前端还会保存 `parsedSourceText`。如果用户继续修改左侧描述，当前文本不再对应右侧草稿，页面会标记“待重新解析”并禁用确认；重新解析后才恢复。编辑已有流水时会显示正在编辑的账单，并提供“取消编辑”，取消只重置前端状态，不调用更新接口。

## 三种 AI 状态

- `source=model, provider=primary|backup`：真实模型返回并通过校验。
- `source=local_rule, provider=local`：没有有效模型配置，主动使用本地规则。
- `source=error_fallback, provider=fallback`：所有已配置 Provider 都调用失败，临时回落本地规则。

界面按本次真实响应显示来源，不能只根据“配置了 Key”就声称正在使用模型。

## 本地金额规则如何避免误判

本地兜底不是取第一个数字，也不是随机语录。金额按优先级识别：

1. 明确单位或符号：`50元`、`50块`、`¥50`。
2. 金额语境：`花了50`、`收入2000`、`报销80`、`共50`。
3. 去掉日期、时间和计量数量后，只剩一个数字时才采用。
4. 仍有多个候选时返回金额缺失，不进行猜测。

例如：

```text
7月11日买了3杯咖啡，微信花了50元
```

系统识别 5000 分，而不是 700 分或 300 分。没有“花了50元”时，日期和杯数会被排除，金额进入 `missing_fields`。

本地规则中的金额正则目前只保证阿拉伯数字。线上 Groq 模型已经实测把“两千元”识别为 200000 分，但这属于模型能力，不应口头包装成本地兜底能力。答辩演示优先使用阿拉伯数字输入，汉字金额可以作为额外展示。

## 为什么收入和支出不能在一句话里直接抵消

当前 `POST /api/ai/parse-transaction` 一次只返回一笔账。像“股票收益两千元，健身房办卡一千元”实际包含两项经济事件，正确数据应该是收入 2000 元和支出 1000 元两笔流水，再由月度统计得出净额 1000 元。

若直接存成一笔收入 1000 元，收入、支出、分类占比和预算都会失真。因此第一版要求一句话对应一笔交易；多笔批量解析需要把返回协议升级为数组并让用户逐笔确认，不在当前版本中假装支持。金额框中的灰色 `50.00` 是占位提示，不代表模型识别结果。

## 周度流水如何划分

周统计固定使用周一到周日。前端的 `type="week"` 选择器先换算出 ISO 周起止日期，再并行读取日期范围内的流水和 `/api/stats/weekly` 汇总。后端重新计算周边界，不信任前端直接提交的总额。

跨月周不会按月底切断。例如 6 月 29 日到 7 月 5 日属于同一个统计周，收入、支出、净额和笔数都覆盖完整七天。行为测试还在前一个周日和后一个周一放置记录，确认边界外数据不会混入。

## 分类、账户和备注怎么处理

- 分类使用关键词字典，例如咖啡、奶茶归为“饮品”，地铁、公交归为“交通”。
- 账户识别微信、支付宝、银行卡和现金。
- 收入关键词决定 `type=income`。
- 备注会删除金额、时间、账户和常见记账动词，保留消费场景。
- 模型返回 `books`、`alipay` 等别名时会映射为“学习”“支付宝”。
- 模型返回未知分类或账户时回落到本地结果，不允许任意值进入下拉框。
- 归一化完成后重新计算缺失字段；金额不大于 0、分类无法识别、账户未指定或时间非法都会与模型自报的 `missing_fields` 合并去重。

## 置信度是什么意思

本地规则识别出金额、分类和账户时置信度为 0.86；缺字段时为 0.52，并列出 `missing_fields`。模型可以返回自己的置信度，但无论多高都需要用户确认。

置信度不是服务可用性指标，也不是用户可以盲信的准确率。低置信度不会触发备用模型；只有请求失败、响应异常或校验失败才会切换 Provider。界面优先提示缺少哪些字段和“手动确认入账”，而不是突出百分比。

## AI 财务点评如何实现

`GET /api/ai/monthly-advice` 只读 SQLite 缓存，不调用模型；`POST /api/ai/monthly-advice` 才把本月收入、支出、预算、分类、账户、趋势和最近流水作为统计 JSON 发给 Provider 链。角色 Prompt 要求：

- 面向学生和年轻用户。
- 只能依据提供的统计，不编造金额。
- 输出 `headline`、`detail` 和 `action_items`。
- `sharp` 可以直接但不能羞辱；`warm` 要温和、具体。

没有模型时，本地分析会根据预算是否设置、使用率、最高分类、主要账户、日均支出和剩余额度生成同结构结果。它是确定性分支，不是随机抽取语录。

## 为什么点评改为手动生成

点评是耗时且可能计费的派生结果，没有必要在用户切页、刷新或每次入账时重复生成。当前策略是：

- 只有进入总览/预算页时读取后端缓存，其他页面连缓存接口都不请求。
- 只有用户点击“生成点评/重新分析”时才调用外部 Provider。
- 成功结果按月份和语气写入 SQLite，总览与预算页共用同一份结果。
- 使用“统计数据 + 语气 + 模型配置 + Prompt 版本”的 SHA-256 指纹判断是否过期，指纹不包含 Key。
- 数据改变后不立刻花一次调用，而是保留旧点评并标记“待更新”。

这个设计的产品权衡是：放弃“每次数据变化都自动实时”，换取调用成本可控、状态可解释和旧结果可追溯。

## API Key 和设置页边界

本地开发：

- `RUNTIME_AI_SETTINGS_WRITABLE=true`。
- 设置页调用 `PUT /api/settings/ai`，配置保存在本地 SQLite。
- Key 输入框留空表示保留旧值。

公开演示：

- `RUNTIME_AI_SETTINGS_WRITABLE=false`。
- `PUT /api/settings/ai` 返回 403。
- Key 只保存在 Render 环境变量。
- 设置页只读展示模型、配置状态和连接测试。

`GET /api/settings/public` 和 `POST /api/settings/ai/test` 都不返回真实 Key。CORS 只能限制浏览器来源，不等于身份认证，因此无登录的公开演示必须关闭配置写入。

## 为什么当前线上没有启用备用模型

主备机制仍然存在于代码和测试中，但容灾不是“接口越多越安全”。调用链是串行的，已知慢备用会在主模型失败后继续占用完整超时时间。

最终诊断中，Render 基础接口约 170–192 ms，Agnes 在 45.2 秒后超时，硅基流动成功也曾需要 21.1 秒；这证明瓶颈在 Render Oregon 到亚洲 Provider 的链路或模型服务，而不是用户浏览器。迁移 Groq 后，Provider 测试为 917 ms，普通快记为 3.64 秒。因此公开演示选择 Groq 主模型、10 秒超时和本地兜底，备用槽位只在第二个低延迟 Provider 通过真实快记测试后启用。

Groq 模型 ID 必须写完整命名空间 `qwen/qwen3.6-27b`。误写成 `qwen3.6-27b` 会得到 404 `model_not_found`；无效 Key 通常是认证错误，二者不能混为一谈。

快记成功并不代表长输出一定成功。首次线上点评验证连续回落本地，原因是 Qwen 长任务可能启用推理或在 JSON 外附带包装文本。最终请求显式设置 `reasoning_effort=none`，并用标准库 `json.JSONDecoder` 从响应中提取合法对象。修复后月度点评在 5.25 秒返回 `source=model, provider=primary`。当前完整基线为 29 项后端行为测试、5 项 Playwright 测试和前端生产构建通过。

## API 挂了怎么办

模型 API 挂掉时，FastAPI 本身仍可用：

- 配置了合格备用 Provider 时，主模型失败后继续尝试备用模型。
- 当前线上 Groq 失败或超时后直接返回本地规则草稿。
- 用户仍可手动填写并保存账单。

如果整个 Render 服务正在休眠，前端不会先显示虚假的零数据。超过约 1.2 秒会提示“连接时间比平时稍长”，网络失败后提供重新加载。GET 请求最多自动重试一次；POST、PUT、DELETE 不自动重试，避免重复入账。

## 非法输入会不会让系统崩溃

- 自然语言为空或超过长度时由 Pydantic 返回 422。
- 金额输入只允许两位小数，非法值在输入框旁提示。
- 交易金额不能为 0、负数或超过产品上限；预算不能为负数。
- 月份必须是 `YYYY-MM` 且月份在 `01–12`，日期范围不能倒置或超过 366 天。
- 类型只能为 `expense` 或 `income`。
- 标签去空、去重，最多 8 个，每个最多 16 字。
- 删除不存在的账单返回 404。
- 前端把 422、网络失败和超时转换为中文提示。

## 演示数据为什么不会自动复活

演示数据由 `SEED_DEMO_DATA` 显式控制。开启后只在空数据库第一次初始化时写入，并在 `app_settings` 记录 `demo_seed_completed=true`。以后即使用户删除全部交易，服务重启也不会再次插入。公开演示可以开启，个人账本默认关闭。

## Provider 失败如何排查且不泄露隐私

AI 调用日志只记录操作类型、请求追踪 ID、主/备槽位、模型名、耗时、HTTP 状态和异常类型，并记录是否进入兜底。日志不写 API Key、Authorization Header、完整账单原文或完整模型响应。这样既能判断是超时、HTTP 错误还是结构校验失败，又不会把用户流水放进服务器日志。

## Render 构建 Bug 怎么解决

Render 默认 Python 3.14 时，固定版本的 `pydantic-core` 没有匹配 wheel，pip 尝试通过 Rust/maturin 源码构建，并因只读缓存目录失败。项目通过 `.python-version` 和 `PYTHON_VERSION=3.11.9` 锁定已验证版本，而不是无目的升级依赖。
